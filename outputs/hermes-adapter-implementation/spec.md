## Overview

This document specifies the Hermes adapter for the Zend home-miner daemon. The adapter provides AI agents ("Hermes") with a capability-scoped interface to read miner status and append summaries to the event spine. Hermes agents cannot issue control commands or read user messages.

## Architecture

```
Hermes Gateway (external AI agent)
      |
      | (HTTPS + authority token)
      v
Zend Hermes Adapter  ←  services/home-miner-daemon/hermes.py
      |
      v
Zend Gateway Contract → Event Spine (spine.py)
```

## Capability Model

### Hermes Capabilities (Milestone 1)

| Capability | Description | Boundary |
|------------|-------------|----------|
| `observe` | Read miner status snapshot | ✅ Allowed |
| `summarize` | Append summaries to event spine | ✅ Allowed |
| `control` | Issue miner start/stop/mode commands | ❌ Blocked |
| `user_message` | Read user messages | ❌ Blocked |

### Event Visibility

Hermes can read these event kinds (defined in `spine.py` `EventKind`):

- `hermes_summary` — Hermes's own summaries
- `miner_alert` — System alerts
- `control_receipt` — Control command receipts

Hermes **cannot** read:
- `user_message` — User messages (filtered out in `hermes.py:get_filtered_events`)

## Token Format

Authority tokens encode Hermes's delegated capabilities:

```
<hermes_id>:<capabilities>:<expiration_iso>
Example: hermes-001:observe,summarize:2027-03-23T00:00:00+00:00
```

Tokens are generated by `generate_authority_token()` and validated by `connect()`.

## Adapter Interface (hermes.py)

### Core Functions

| Function | Capability Required | Description |
|----------|-------------------|-------------|
| `pair_hermes(hermes_id, device_name)` | None | Create idempotent pairing record |
| `generate_authority_token(hermes_id)` | None | Generate authority token for paired Hermes |
| `connect(authority_token)` | None | Validate token, return HermesConnection |
| `read_status(connection)` | `observe` | Read miner status snapshot |
| `append_summary(connection, summary_text, authority_scope)` | `summarize` | Append hermes_summary event |
| `get_filtered_events(connection, limit)` | None | Read events (user_message filtered) |
| `get_hermes_pairing(hermes_id)` | None | Retrieve pairing record |
| `check_hermes_auth(hermes_id)` | None | Check if pairing is valid |

## Daemon Endpoints (daemon.py)

| Endpoint | Method | Auth | Description |
|----------|--------|------|-------------|
| `/hermes/pair` | POST | None | Create Hermes pairing |
| `/hermes/token/<hermes_id>` | GET | None | Generate authority token |
| `/hermes/connect` | POST | None | Connect with authority token |
| `/hermes/status` | GET | Hermes | Read miner status |
| `/hermes/summary` | POST | Hermes | Append summary |
| `/hermes/events` | GET | Hermes | Read filtered events |
| `/hermes/disconnect` | POST | Hermes | Disconnect Hermes |
| `/hermes/pairing/<hermes_id>` | GET | None | Get pairing status |

Hermes auth header format: `Authorization: Hermes <hermes_id>`

Control endpoints (`/miner/start`, `/miner/stop`, `/miner/set_mode`) reject Hermes auth with 403.

## CLI Commands (cli.py)

```bash
# Pair Hermes agent
python cli.py hermes pair --hermes-id hermes-001 --device-name "Living Room Miner"

# Generate authority token
python cli.py hermes token --hermes-id hermes-001

# Read miner status via Hermes adapter
python cli.py hermes status --hermes-id hermes-001

# Append summary
python cli.py hermes summary --hermes-id hermes-001 --text "Miner running normally" --scope observe

# Read filtered events
python cli.py hermes events --hermes-id hermes-001 --limit 20
```

## Security Boundaries

1. **Capability Enforcement**: Each operation checks for required capabilities before execution
2. **Event Filtering**: `user_message` events are excluded from `get_filtered_events()`
3. **Token Validation**: Authority tokens validated for format, expiration, capability scope
4. **Control Blocked**: Control endpoints reject Hermes auth with 403
5. **Pairing Isolation**: Each Hermes pairing has a unique principal_id
